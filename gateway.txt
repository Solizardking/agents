1. ARCHITECTURE OVERVIEW
──────────────────────────────────────────────────────────────────────────

The CLAWD Gateway (`gateway/src/agentRegistry.ts`) serves the full 124+ agent
catalog from the `agents/` directory. When a user runs `install.sh --gateway`,
they get:

  • Telegram bot  —  /menu, /wallet, /trade, /alerts, x402 tier gating
  • HTTP API      —  Express server on port 8080 (or GATEWAY_PORT)
  • Agent catalog —  Static JSON served from `agents/agents-catalog.json`
  • Agent registry—  Individual agent JSONs from `agents/public/api/agents/`
  • Templates     —  agent-template.json, attested, full
  • ACP discovery —  .well-known/ai-plugin.json, .well-known/acp.json
  • MCP endpoints —  modelcontextprotocol discovery

The gateway resolves the repo root by walking up from `gateway/src/` until it
finds `agents/agents-catalog.json`. No secrets, API keys, or credentials
are exposed — it reads only static JSON files from disk.

3. INSTALL FLOW — HOW USERS GET THE GATEWAY
──────────────────────────────────────────────────────────────────────────

  # Full install (everything)
  curl -fsSL https://raw.githubusercontent.com/solizardking/solanaclawd/main/install.sh | bash -s -- --full

  # Gateway only
  curl -fsSL https://raw.githubusercontent.com/solizardking/solanaclawd/main/install.sh | bash -s -- --gateway

  # Or clone and run manually
  git clone https://github.com/solizardking/solanaclawd
  cd solanaclawd
  bash install.sh --gateway

The --gateway flag in install.sh does:
  1. cd gateway && npm install
  2. npm run build (tsc)
  3. Ready to start: cd gateway && npm start

When the gateway starts:
  • agentRegistry.ts resolves REPO_ROOT → finds agents/agents-catalog.json
  • Express mounts all catalog/registry/template endpoints
  • Telegram bot starts (webhook or long-polling mode)
  • Birdeye WebSocket connects for price alerts
  • /health reports status


4. HOW THE GATEWAY DISCOVERS THE CATALOG
──────────────────────────────────────────────────────────────────────────

The gateway does NOT use hardcoded paths. It uses a `findRepoRoot()`
function that walks up from `gateway/src/` looking for `agents/agents-catalog.json`:

  gateway/src/agentRegistry.ts (compiled from gateway/src/ directory)
    → ../../ (repo root)
    → checks agents/agents-catalog.json
    → if found: all endpoints work
    → if not found: graceful fallback, returns empty arrays

This means the gateway works in any deployment layout as long as the
`agents/` directory is a sibling of `gateway/`:

  ✓  /app/agents/agents-catalog.json  +  /app/gateway/src/index.ts
  ✓  /repo/agents/...                  +  /repo/gateway/...

Set GATEWAY_BASE_URL to control the base URL in responses. Default is
https://x402.wtf.


5. SECURITY — WHAT IS AND IS NOT EXPOSED
──────────────────────────────────────────────────────────────────────────

  EXPOSED (public, intentional):
    • All agent catalog JSONs (src/, public/api/agents/)
    • Catalog metadata (agents-catalog.json)
    • Agent templates
    • ACP/.well-known discovery documents
    • /health, /identity, /registry endpoints

  NOT EXPOSED (protected):
    • No .env files (gitignored, never read by agentRegistry)
    • No private keys (SOLANA_PRIVATE_KEY used ONLY by solana.ts for signing)
    • No API keys (HELIUS_API_KEY, BIRDEYE_API_KEY, ANTHROPIC_API_KEY
      used ONLY by their respective modules, never in agentRegistry)
    • No session data (sessionStore.ts reads from /data/sessions.json,
      not served via HTTP)
    • No wrangler/cloudflare state (.wrangler/ is gitignored)
    • No Rust target/ directories (gitignored)
